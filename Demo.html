Verdict: Engine Exists. Experience Missing
AppScan 360° has the data, the correlation engine, and the AI fix capability — but they are scattered across separate pages and limited to specific scan types (e.g. RapidFix is SAST-only; correlation requires IAST). This is what needs to be developed .

How Developer Ownership Mapping Is Fetched — Git + Jira Integration
Step 1 — Git
Module → Repository mapping
Each AppScan issue carries a file location path (e.g. src/user-service/auth.js). The dashboard reads the repo's CODEOWNERS file via GitHub/GitLab API to match that path prefix to the owning team.
Step 2 — Jira
Module → Team name mapping
Jira project components each have a lead team. The dashboard calls /rest/api/3/project/{key}/components to build a component → team index and optionally correlate open security tickets by CVE/CWE label.
Step 3 — AppScan
Issue → Module → Owner resolution
AppScan location path → matched against Git CODEOWNERS → resolved to team name from Jira component → aggregated as open issues, critical count, and avg age per module in the ownership table.
Prerequisite: Requires one-time admin configuration of the module → Git repo → Jira project → team mapping. Accuracy depends on CODEOWNERS files and Jira component leads being maintained. This is a data quality dependency, not a technical blocker.

Method Endpoint Purpose Used for

Method Endpoint Purpose & notes Used for
GET /repos/{org}/{repo}/contents/CODEOWNERS Reads the CODEOWNERS file to resolve file path prefix → owning team. Base64-decoded response parsed line by line. Path → team resolution

How AI Fix Advisor Routes Your Request — Per-Issue, Per-Scan-Type
Step 1 — Context Extraction
Issue → Structured Prompt
The dashboard reads the selected AppScan issue: scan type (SAST/DAST/IAST/SCA), CWE ID, severity, location path, and finding description. These are assembled into a structured prompt payload — no raw source code is included.
Step 2 — Backend Routing
Prompt → Selected LLM Endpoint
The prompt is dispatched to the customer-configured backend: Claude → api.anthropic.com/v1/messages Ollama → localhost:11434/api/generate OpenAI-compat → {host}/v1/chat/completions
Step 3 — Response Rendering
LLM Output → Fix Advisor Panel
The LLM response is parsed and rendered as: plain-English vulnerability explanation, code fix with before/after diff, business impact summary, CWE reference, and a one-click "Create Jira Ticket" action — across all scan types.
Step 4 — Fix Simulation
Pre-Fix → Risk Score Preview
After AI generates the fix, the dashboard re-evaluates the security debt score with the issue resolved — showing the predicted drop in risk score before any code is committed. Helps teams prioritise fixes by impact.
Data privacy guarantee: Only issue metadata (CWE ID, scan type, severity, location path, description) is sent to the AI backend. No source code, no credentials, no customer PII ever leaves the environment. For Ollama deployments, all inference runs fully local — zero external data transmission.
Backend Method Endpoint Best suited for
Claude (Anthropic) POST api.anthropic.com/v1/messages Highest fix quality; official, stable API with clear data handling policy
Ollama (Local LLM) POST localhost:11434/api/generate Air-gapped / zero data exposure environments. Runs llama3, mistral, codellama locally.
OpenAI-compatible POST {host}/v1/chat/completions Azure OpenAI, vLLM, LM Studio, or any OpenAI-compatible endpoint
GitHub Copilot — not a standalone backend option
GitHub Copilot has no official public chat completions API outside of IDE extensions. Adding it as a first-class backend like Claude or Ollama is not viable without unofficial reverse-engineered proxies (which carry ToS risk and no stability guarantees). If your org uses Copilot Enterprise via Azure OpenAI — your Azure OpenAI endpoint is already fully compatible with the OpenAI-compatible slot above. That is the supported path for Copilot Enterprise customers.

How External Enrichment Data Is Applied — EPSS + CVE Intel Pipeline
Step 1 — CVE Extraction
AppScan SCA Issue → CVE ID
For each SCA finding, the dashboard extracts the CVE identifier from the AppScan issue payload (e.g. CVE-2024-23917). SAST and DAST findings without a CVE are enriched via their CWE ID for classification and clustering instead.
Step 2 — EPSS Lookup
CVE → Exploit Probability Score
The dashboard calls api.first.org/data/v1/epss?cve={id} to fetch the EPSS score (0–1 probability) and percentile, updated daily by FIRST.org. Results are cached per session to avoid redundant calls for repeated CVEs across multiple findings.
Step 3 — Priority Score Overlay
EPSS → 5-Factor Risk Score
EPSS score feeds into the 5-factor priority formula:
Severity×40% + EPSS×25% + Asset×20% + Exposure×10% + Data×5% The weighted score drives bubble chart placement, issue table sort order, and AI fix advisor prioritization across all scan types.
No authentication required: Both FIRST.org EPSS API and the Chart.js CDN are fully public. The enrichment pipeline adds zero credential management overhead. EPSS data is fetched client-side on demand — no backend proxy needed, no data stored server-side.
Method Endpoint Data provided Used for
GET api.first.org/data/v1/epss?cve=CVE-XXXX EPSS score (0–1) and percentile for a given CVE — the probability it will be exploited in the wild within 30 days. Updated daily by FIRST.org. Priority scoring, bubble chart, issue table column
GET cdnjs.cloudflare.com — Chart.js 4.4.1 Client-side charting library loaded via CDN. Powers all bar, donut, line, and bubble charts in the dashboard. No server dependency. All dashboard charts and visualizations

Implementation Steps — How to Deploy
From zero to live dashboard in 6 steps
Step 1 — Deploy the Dashboard
Open appscan_unified_dashboard.html in Chrome/Edge. No install, no build, no server needed. For team access, host on any static file server (Nginx, S3, Netlify, IIS) or share via corporate intranet.
Step 2 — Connect to AppScan 360°
Enter your AppScan 360° base URL and API key in the dashboard config banner. The dashboard calls GET /api/v4/Issues to pull all SAST/DAST/IAST/SCA issues. v2.1.0 auth: header X-Api-Key: {KeyID}:{KeySecret} — create at Admin → API Keys → Create. 30-min auto-refresh session. Service accounts also supported.
Step 3 — Configure AI Backend
Choose one of three AI providers: Claude API (paste Anthropic API key), Local LLM (install Ollama: curl -fsSL https://ollama.com/install.sh | sh && ollama pull llama3:8b), or OpenAI-compatible (enter endpoint + key). Click "Test" to verify connectivity.
Step 4 — Configure Gate Policies
Define pass/fail rules for Production, Staging, and Development gates. Gate evaluator auto-checks: zero critical issues, no CVSS ≥9.0 in SCA, OWASP compliance, IAST coverage thresholds, license approvals. Override workflow available for break-glass scenarios.
Step 5 — Enrich with EPSS
Dashboard auto-queries api.first.org/data/v1/epss?cve=CVE-XXXX for issues with CVEs across all scan types (SAST/DAST/IAST/SCA) to overlay real-world exploitability scores. Bubble chart and table column display EPSS values for prioritization.
Step 6 — Triage & Remediate
Click any issue row to open the AI Fix Advisor. Choose: Generate Code Fix (before/after diff), Explain Vulnerability (plain English), OWASP Reference, or Business Impact assessment. Export filtered issues as CSV/JSON/PDF.
